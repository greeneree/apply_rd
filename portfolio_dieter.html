프로젝트 개요

배경

MAU 24만, 누적 다운로드 100만+ 규모의 다이어트 앱. 캐릭터 IP 계약 만료로 인해 서비스 정체성 전면 재정립이 필요한 상황에서 PM으로 합류. 단순 리스킨이 아닌 사용자가 지속적으로 앱에 복귀할 이유를 구조적으로 제공하는 것을 목표로 삼았다.

핵심 미션

캐릭터 의존 구조를 탈피해 콘텐츠와 행동 습관 기반의 플랫폼으로 전환. 공공 식품 칼로리 API 연동, 운동 콘텐츠 자체 제작 파이프라인 구축, 데일리 미션 Gamification 설계. 오프라인 유저 세미나와 뷰티·다이어트 체험단 광고를 통한 커뮤니티 운영까지 End-to-End 수행.

데이터가 드러낸 구조적 문제
30%
전체 유저 중 만렙 도달 비율 (GA 분석)

MAU 24만 중 약 7만 명 이상이 이미 게임 구조의 끝을 본 상태였다. 단순히 콘텐츠 업데이트 속도 문제로 봤는데, 만렙 이후에도 잔류하는 유저들이 히든맵에 집중된다는 걸 발견하면서 해석이 바뀌었다. 사용자는 끝이 있는 구조보다 예측 불가능한 보상에 더 반응하고 있었다.

체중 감량 앱의 이탈 원인은 "의지 부족"이 아니라 "구조의 문제"다.
체중 1kg 감량보다, "이번 주 샐러드 3번 먹었다"는 작은 성공의 누적이 커뮤니티 응원과 결합할 때 더 긴 retention을 만든다.
Gamification 설계 — 정량에서 자기효능감으로
만렙 고인물 30% 속출
→
끝없는 수치 구조로 전환
→
행동 기반 미션 설계
→
커뮤니티 응원 루프
→
자기효능감 → retention
⚠ 미검증 가설

이 구조는 2.0 기획 단계에서 설계되었으나, 서비스 종료(옐로모바일 투자 철회)로 인해 실제 사용자 반응을 검증하지 못했다.

단, 이 가설이 지금도 유효하다고 생각한다. 체중 감량 앱의 이탈 원인이 "의지 부족"이 아니라 "구조의 문제"라는 판단은 변하지 않았다. 그리고 이건 단순한 이론이 아니다 — 나는 40kg을 직접 감량했고, 요요도 겪었다. 뺄 때 앱이 재밌었고, 정체기가 오자 앱부터 지웠다.

문제 해결
01

캐릭터 IP 만료 → 서비스 정체성 재설계

상황

캐릭터 IP 계약 만료로 앱 전면 개편 필요. 단순 리스킨 시 서비스 차별화 요소 소멸 위기.

분석

IP 만료가 오히려 구조 재설계의 명분. 캐릭터 의존이 아닌 콘텐츠·습관 형성 구조로 전환 필요.

결과

운동 콘텐츠·칼로리 DB·데일리 미션 3축 구조로 재편. 사용자 재방문 루프 설계 완성.

02

콘텐츠 부재 → 자체 제작 파이프라인 0→1

상황

신뢰할 수 있는 운동 콘텐츠 확보 필요. 외부 라이선스 비용 부담.

분석

공공 식품 칼로리 API로 DB 비용 절감 가능. 트레이너 직접 섭외로 자체 콘텐츠 제작 시 비용 대비 신뢰도 확보.

결과

공공 API 기반 칼로리 DB 연동 + 트레이너 섭외 운동 영상 자체 제작. 콘텐츠 파이프라인 0→1 완성.

03

Waterfall 일괄 개편 → 일정 지연 · 팀 소진

상황

서비스 전체를 한꺼번에 개편하는 방식 선택. 오픈 일정 약 1개월 지연 발생.

분석

기능을 더 줄이고 MVP로 먼저 오픈했다면 사용자 반응을 한 달이라도 더 빨리 볼 수 있었다. 결과보다 검증 속도가 먼저여야 한다.

결과

이후 모든 프로젝트에서 2주 스프린트·MVP 검증 우선 방식으로 전환. 직방·군장병 등 이후 프로젝트에 적용.

Lesson Learned

✓ 유지해야 할 것

CX·오프라인 세미나로 유저 페인포인트 직접 수집 — 앱 개선 반영 사이클의 핵심
공공 API 활용 — 콘텐츠 신뢰도와 비용 효율 동시 확보
결과(체중)가 아닌 과정(습관)을 보상하는 Gamification 구조
오프라인 커뮤니티 연계로 앱 재방문 루프 강화

⚠ 보완해야 할 것

MVP 우선 — Waterfall 방식 탈피. 빠른 오픈이 빠른 학습보다 중요하다
가설 검증을 기획 초기에 — 만렙 구조 문제는 더 일찍 발견 가능했다
미검증 가설을 놓지 않기 — 자기효능감 기반 retention 구조는 언젠가 검증할 것
"체중 감량 앱의 이탈은 의지 부족이 아니라 구조의 문제다"

다이어터 앱에서 배운 것: 사용자는 목표(체중감량)를 원하지 않는다. 오늘 하루 작은 성공 경험을 원한다.

칼로리 DB + 데일리 미션 + 기록 루프가 이 구조를 만들려던 방법이었다.
그리고 소프트웨어만으로는 헬스케어 서비스의 진짜 우위를 점령하기 어렵다는 걸 그때 처음 확신했다.
디바이스와 연동된 혈당·혈압 데이터가 있어야 한다는 생각도 그때부터였다.
